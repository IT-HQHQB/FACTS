1. Document Overview

1.1 Purpose

This System Architecture and Design Document (SAD) is the authoritative technical reference for the FACTS (File and Case Tracking System). It is intended to communicate structural decisions, design rationale, and operational guidelines to engineers, architects, administrators, and technical stakeholders involved in building, deploying, maintaining, or extending the system.

The document covers end-to-end architecture including logical decomposition into modules, physical deployment topology, data modeling, security controls, integration patterns, performance targets, and operational procedures.

1.2 Scope

This document covers the complete FACTS platform deployed for the Baaseteen and SHND welfare organizations. System scope includes:

  • Backend REST API (Node.js / Express.js) — 150+ endpoints across 25 route modules
  • Frontend Single-Page Application (React 18) — 31 page components with role-adaptive UI
  • MySQL 8.0 relational database — 60+ normalized tables
  • File storage for case attachments, digital signatures, photos, and generated PDFs
  • Integration with the external ITS (Institutional Information System) API
  • Optional SMTP email notification pipeline

Out of scope: Third-party ITS platform internals, SMTP relay infrastructure, and any future mobile application layer.

1.3 Assumptions and Constraints

Assumptions
  • Node.js v16+ and MySQL v8.0+ available in target environment
  • External ITS API accessible over HTTPS with valid credentials
  • Users access via modern web browser (Chrome, Firefox, Edge)
  • File uploads do not exceed 10 MB per file
  • Concurrent users per deployment ≤ 500
Constraints
  • Jamiat/Jamaat organizational scoping required for data isolation
  • JWT tokens expire after 24 hours; no refresh token in v1
  • PDF generation requires Puppeteer (Chromium binary dependency)
  • MySQL connection pool capped at 10 concurrent connections per instance
  • Base64-encoded photos stored directly in DB (no object storage in v1)

2. System Overview

2.1 System Description

FACTS is a comprehensive welfare case management platform designed to digitize and govern the full lifecycle of financial assistance cases for Baaseteen and SHND organizations. It replaces paper-based counseling forms, manual approval chains, and fragmented tracking spreadsheets with a unified, role-aware digital workflow.

A case begins with an eligibility screening (Case Identification), progresses through a structured counseling phase, then traverses a multi-tier approval pipeline — Welfare Review → ZI Approval → KG Review → Operations Lead → Cover Letter Generation → Executive Approval → Manzoori Sanction — before reaching Financial Disbursement and periodic Fund Utilization tracking, ultimately closing with documented evidence of aid completion.

2.2 System Context

External: ITS API

Institutional Information System. Provides applicant biographical data, photos, Jamiat/Jamaat affiliations by ITS number. Read-only HTTPS REST integration.

FACTS Platform

Core system: Express.js API + React SPA + MySQL. Manages all case data, workflows, approvals, documents, notifications, and reports.

External: SMTP Relay

Optional email delivery (Gmail SMTP or equivalent). Receives outbound notification payloads from the FACTS notification service.

User Roles: Super Admin, Admin, DCM (Deputy Counseling Manager), Counselor, Welfare Reviewer, ZI (Zonal Incharge), KG Reviewer, Operations Lead, Executive Management, Finance Officer, Report Viewer

3. Architecture Goals and Principles

3.1 Goals

Correctness & Auditability
Every state transition, approval, and data edit is logged. Status history, workflow comments, and income audit logs provide a complete audit trail.
Role-Based Access Control
All resources protected by a fine-grained permission model (resource + action). Organizational scoping (Jamiat/Jamaat) further restricts data visibility.
Workflow Configurability
Approval stages, role assignments, SLA thresholds, and checklist items are database-driven — configurable without code changes.
Performance & Responsiveness
API responses within 300ms for paginated list endpoints, 500ms for complex aggregations under normal load.
Data Integrity
Foreign-key constraints, parameterized queries, and transactional operations protect relational integrity across 60+ tables.

3.2 Principles

Separation of Concerns
API routes, business logic, and database access are structurally separated. Frontend pages, components, and contexts independently organized.
Fail Safely
Errors return structured JSON with meaningful status codes. Unhandled exceptions caught by global middleware without exposing stack traces to clients.
Least Privilege
Tokens carry minimal claims. Every request revalidates permission and organizational scope against the live database.
Minimal External Dependencies
System avoids cloud-vendor lock-in. Core functionality runs on any Linux server with Node.js and MySQL.
Progressive Enhancement
System degrades gracefully if optional services (SMTP, Puppeteer) are unavailable; core case management remains functional.

Serves React build (static files), terminates SSL, reverse-proxies /api and /uploads to Node.js on port 5000.

Single Express process managed by PM2. Listens on port 5000 (localhost only). Cluster mode available for multi-core servers.

Same host for small deployments or dedicated VM. Uses InnoDB engine; connection pool size: 10.

Local ./uploads/ directory. Served via Express static middleware at /uploads/*.

443 (HTTPS), 80 (HTTP → HTTPS redirect). All other ports closed by firewall.

5000 (Node.js API, localhost only), 3306 (MySQL, localhost only).

Nginx handles TLS 1.2+. Certificates via Let's Encrypt (certbot) or organization CA.

Outbound HTTPS to ITS endpoint. Must be whitelisted in egress firewall rules.

4.3 Process View — Operations & Performance

4.3.1 Request Lifecycle

Every HTTP request passes through the following Express middleware chain:

  1. CORS — Validates Origin against FRONTEND_URL
  2. Helmet — Sets security-related HTTP response headers
  3. express-rate-limit — Per-IP rate limit on /api/auth/login
  4. express.json() — Parses JSON request bodies
  5. auth middleware — Verifies JWT, loads user from DB, checks is_active
  6. permission middleware — Validates resource:action and Jamiat/Jamaat scope
  7. Route handler — Business logic and DB queries
  8. Error handler — Catches thrown errors, returns structured JSON

4.3.2 Concurrency & Connection Management

MySQL Connection Pool Configured with a maximum of 10 simultaneous connections. Requests queue when the pool is exhausted. For higher concurrency, increase the pool size in config/database.js and align MySQL server's max_connections accordingly.

Node.js handles requests asynchronously via the event loop. All database operations use async/await with mysql2/promise. File I/O (Multer uploads, PDF generation) is non-blocking via Node's libuv thread pool.

4.3.3 Error Handling & Fault Tolerance

Error Class Handling Strategy Client Response
Validation errors Joi schema validation before DB access 400 + field-level error message
Auth failures JWT verify + active user DB check 401 / 403 JSON error
DB constraint violations mysql2 error code inspection 409 / 400 JSON error
File upload errors Multer error handler middleware 400 + Multer error message
PDF generation failure try/catch around Puppeteer/PDFKit 500 + "PDF generation failed"
External ITS API timeout Axios timeout + catch, fallback to DB 503 or cached DB data
Unhandled exceptions Global Express error middleware 500 + generic message (no stack trace)

4.4.3 Coding Conventions

  • All backend queries use parameterized statements via mysql2 — no dynamic SQL string construction
  • Frontend components use functional style exclusively (no class components); state managed via hooks

5.3 Schema Design Patterns

  • Soft deletes: Roles, workflow stages, and checklist items use is_active boolean to preserve referential integrity and audit history rather than physical deletion.
  • JSON columns: workflow_history (cases), associated_statuses (workflow_stages), responses_json (fund_utilization_reports) store variable-structure data without schema migrations.
  • Base64 photos: Stored as longtext in database (v1 constraint). Recommendation: migrate to object storage path reference in v2 for scale.
  • Audit logging: case_identification_income_logs captures before/after values for income edits. status_history captures every case status transition with actor and timestamp.
  • SLA fields: workflow_stages.sla_value, sla_unit, sla_warning_value, sla_warning_unit drive the automated SLA calculator per stage.
  • Dynamic forms: fund_utilization_sections and fund_utilization_questions allow admin-configurable form structure without code changes.

6. Security Architecture and Design

6.1 Authentication & Authorization

Authentication Flow
  1. User submits credentials to POST /api/auth/login
  2. bcryptjs compares password against stored hash (10 salt rounds)
  3. On success: signed JWT issued with { userId, role, iat, exp }
  4. Token lifetime: 24 hours (configurable via JWT_EXPIRES_IN)
  5. Client sends Authorization: Bearer <token> on every request
  6. Auth middleware reloads user from DB and checks is_active = 1 on every request
Authorization Layers

Layer 1 — RBAC: Permissions stored as resource:action pairs in role_permissions. Effective permissions resolved from all assigned roles and injected into req.user.permissions.

Layer 2 — Organizational Scope: user_roles stores jamiat_ids and jamaat_ids per role. Middleware filters case queries to the user's authorized organizations.

Layer 3 — Stage Permissions: counseling_form_stage_permissions controls per-role read/update access at the form section level.

6.2 Data Security

Control Implementation
Password Storage bcryptjs, 10 salt rounds — no plaintext passwords ever stored
Transport Encryption TLS 1.2+ enforced at Nginx layer for all client traffic
Data at Rest MySQL tablespace encryption (AES-256) recommended for production deployments
SQL Injection Prevention 100% parameterized queries via mysql2 prepared statements
Input Validation Joi schema validation on all POST/PUT request bodies
File Upload Security Multer enforces MIME type whitelist and 10 MB per-file size limit
Sensitive Configuration DB credentials, JWT secret, SMTP password in .env — never committed to source control

6.3 Network Security

  • Helmet.js sets Content-Security-Policy, X-Frame-Options, HSTS, and other protective HTTP headers on all API responses
  • CORS restricts Access-Control-Allow-Origin to the configured FRONTEND_URL only
  • Rate Limiting on /api/auth/login via express-rate-limit; trusted IPs whitelisted via LOGIN_RATE_LIMIT_SKIP_IPS
  • Firewall: Only ports 80, 443, and 22 (SSH) externally accessible. MySQL (3306) and Node.js (5000) must be bound to localhost only
  • Egress: ITS API calls are outbound only; credentials stored in environment variables

8. Performance, Scalability, and Optimization

8.1 Performance Targets

Endpoint Type Target Response Time Notes
Authentication (login) < 200 ms bcrypt comparison ~100ms at 10 rounds
List endpoints (paginated) < 300 ms Cases, users, notifications with default limit=10
Single record fetch < 150 ms Case detail, user by ID
Workflow actions < 400 ms Status update + notification creation
PDF generation (cover letter) < 3 s Puppeteer HTML→PDF; Chromium startup overhead
Excel import (100 rows) < 2 s SheetJS parse + batch DB inserts
Dashboard overview < 500 ms Multiple aggregate COUNT queries

8.2 Scalability Design

Vertical Scaling

Increase server CPU/RAM. Node.js benefits from more heap for Puppeteer. MySQL benefits significantly from a larger InnoDB buffer pool (target 70–80% of available RAM).

Horizontal Scaling

PM2 cluster mode: pm2 start index.js -i max spawns one process per CPU core. JWT is stateless so no sticky sessions needed. Requires shared MySQL and shared file storage (NFS or object store).

Database Optimization

Key indexes on: cases.status, cases.jamiat_id, cases.applicant_id, applicants.its_number, notifications.user_id + is_read. All list endpoints paginated — never unbounded SELECT.

Caching Strategy

Frontend: React Query caches server data with configurable staleTime. Master data (case types, Jamiat) refetched infrequently. Backend: No server-side cache in v1; Redis can be added in v2 for session/rate-limit state.

Base64 Photo Storage Warning Storing photos as Base64 longtext inflates row sizes significantly. For deployments with >1,000 applicants, migrate photos to file system paths or S3-compatible object storage and store only the URL in the DB.

9. System Availability and Resilience Design

9.1 Redundancy & Failover

Component Redundancy Strategy RTO Target
Application Server PM2 auto-restart on crash or systemd Restart=on-failure < 30 seconds
Database MySQL primary + read replica replication; manual failover promotion < 5 minutes
Nginx systemd auto-restart; keepalived for dual-Nginx HA if required < 30 seconds
File Storage RAID or replicated NFS for /uploads; daily rsync to backup location < 1 hour (restore)

9.2 Backup & Disaster Recovery

Recommended Backup Schedule Daily full MySQL dump with 30-day retention. Weekly off-site copy. Uploads directory synced daily to secondary storage via rsync or cloud backup agent. 
# Daily MySQL backup cron (example — runs at 2 AM)
0 2 * * * mysqldump -u root -p baaseteen_cms \
  --single-transaction --quick \
  | gzip > /backups/facts_$(date +%F).sql.gz

# Retain last 30 backups, delete older
find /backups -name "facts_*.sql.gz" -mtime +30 -delete

Recovery procedure: restore from latest dump (mysql baaseteen_cms < backup.sql), restore uploads directory, restart Node.js service. Full recovery test recommended quarterly.

  1. Build React frontend: cd frontend && npm run build
  2. Copy build to web root: /var/www/FACTS/frontend/build/
  3. Pull latest backend, install deps: npm install --production
  4. Apply DB migrations: mysql baaseteen_cms < migrations/<file>.sql
  5. Restart application: pm2 restart facts-backend
  6. Verify health: curl https://domain.com/health

10.3 Monitoring & Alerting

Application Monitoring
  • PM2 built-in: pm2 monit for real-time process stats
  • Health endpoint: GET /health returns server status
  • Error logging: stdout/stderr via PM2 log files
  • Recommended: Grafana + Prometheus Node.js exporter for metrics dashboards
SLA Monitoring
  • Built-in SLA calculator computes on_time / at_risk / breached per case stage
  • Admin trigger: POST /api/notifications/check-sla creates alerts
  • Recommended: schedule SLA check cron every 4 hours

10.4 Maintenance Schedule

  • Monthly: npm audit on backend and frontend; apply security patches; OPTIMIZE TABLE on high-write tables (notifications, status_history)
  • Weekly: Review PM2 error logs; check backup completion; verify SSL cert expiry dates
  • Quarterly: Full disaster recovery test; review and rotate JWT secret; test SMTP delivery

12. Technical Architecture for Hosting, Development & Infrastructure

12.1 Hosting Strategy

FACTS is designed as a self-hosted, on-premises or private-cloud application. This ensures full data sovereignty for sensitive welfare case information and avoids recurring cloud vendor costs for small-to-medium scale deployments.

Resource Minimum Specification Recommended

12.2 Infrastructure Configuration

Compute

  • PM2 cluster mode: pm2 start index.js -i max — utilizes all CPU cores
  • Puppeteer requires Chromium: apt-get install chromium-browser
  • Set NODE_OPTIONS=--max-old-space-size=2048 for large PDF generation workloads

Database

  • Enable innodb_file_per_table=1 for easier per-table management
  • Set innodb_buffer_pool_size to 60–70% of server RAM
  • Enable binary logging (log_bin) for point-in-time recovery
  • Create a dedicated application DB user with minimum privileges: SELECT, INSERT, UPDATE, DELETE on FACTS DB only

Storage

  • Upload directory: /var/www/FACTS/backend/uploads/ must be writable by the Node.js process user
  • Mount uploads on a separate logical volume to prevent disk-full events from crashing the OS

Stage Tool Action
Version Control Git (GitHub / GitLab) Feature branches → PR review → merge to main

CD — Staging SSH deploy / rsync action Deploy to staging, run DB migrations, restart PM2
