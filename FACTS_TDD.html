1. Introduction & Purpose

1.1 Purpose

This Technical Design Document (TDD) is the authoritative engineering specification for the FACTS system. It translates the product requirements defined in the PRD v1.0 into precise architectural decisions, data flow diagrams, database schema rationale, module-level implementation patterns, security controls, and deployment topology — providing a complete reference for developers, DevOps engineers, and technical reviewers building or maintaining the system.

1.2 Scope

This document covers all backend services, frontend architecture, database design, external integrations, and infrastructure configuration for FACTS v1.0. It does not re-document product-level requirements (refer to PRD v1.0) or reproduce the full API request/response schemas (refer to API_SCHEMAS.md).

1.3 Design Principles

Principle Implementation
Defense in depth Security enforced at middleware, route, and query layers — never only at UI
Scope isolation by default Every query filtered by Jamiat/Jamaat scope at the middleware layer; unrestricted access is the exception, not the default
Audit everything All state transitions (case status, income edits, workflow actions) are append-only logged with actor + timestamp
Fail gracefully External API failures (ITS) fall back to manual entry; PDF failures surface user-friendly errors; no unhandled Promise rejections
Separation of concerns Routes handle HTTP; services handle business logic; utilities handle cross-cutting concerns (SLA, notifications, permissions)

2.2 Request Lifecycle

Browser → HTTPS Request
  → Nginx (reverse proxy, TLS termination, static file serving)
    → Express app (port 5000)
      → Global middleware (Helmet, CORS, rate-limit, JSON parser)
        → Route match
          → auth.js: verifyToken()
            → JWT.verify(token, JWT_SECRET)
            → DB: SELECT user WHERE id = payload.id AND is_active = 1
            → Attach req.user = { id, role, executive_level, jamiats[], jamaats[] }
          → auth.js: checkPermission(resource, action) [if route requires it]
            → Resolve permissions from user_roles → role_permissions
            → 403 if missing
          → auth.js: applyScopeFilter() [scope-aware routes]
            → Build WHERE clause fragment injected as req.scopeFilter
          → Route Handler
            → Joi input validation
            → Business logic (DB queries via prepared statements)
            → notificationService / emailService (async, non-blocking)
            → JSON response { data } or { error }
      → Error handler middleware → structured log + safe 500 response

4. Database Design

4.1 Conventions & Paradigms

  • Engine: InnoDB on all tables — required for FK constraint enforcement and row-level locking.
  • Collation: utf8mb4_unicode_ci — supports full Unicode, emoji, and Arabic/Urdu/Gujarati characters in applicant names and addresses.
  • Primary keys: INT AUTO_INCREMENT id on all tables. No UUIDs — simpler JOINs and lighter index storage.
  • Naming: All tables and columns use snake_case. No reserved word collisions; no abbreviations except well-known ones (its, dcm).
  • Audit columns: created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, created_by INT FK(users.id), updated_by INT FK(users.id) — present on all entity tables.
  • Soft deletes: Configurable entities (workflow_stages, checklist items) use is_active TINYINT(1) instead of DELETE — preserves historical case data integrity.
  • JSON columns: LONGTEXT used for JSON storage (workflow_history, responses_json, options_json, associated_statuses) — compatible with MySQL 5.7+ JSON functions while allowing large payloads.

5. Core Module Designs

5.1 Authentication & JWT Middleware

Authentication is fully stateless. No server-side session store. Every request must carry a valid JWT in the Authorization: Bearer <token> header. 

// backend/middleware/auth.js  —  verifyToken()
const verifyToken = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Access denied.' });

  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET);

    // Always re-fetch user to catch deactivation after token issue
    const [rows] = await pool.execute(
      'SELECT id, role, executive_level, is_active FROM users WHERE id = ?',
      [payload.id]
    );
    if (!rows[0] || !rows[0].is_active)
      return res.status(401).json({ error: 'User inactive or not found' });

    // Resolve all active roles + Jamiat/Jamaat scopes from user_roles junction
    const [roleRows] = await pool.execute(
      `SELECT r.name, ur.jamiat_ids, ur.jamaat_ids
         FROM user_roles ur JOIN roles r ON r.id = ur.role_id
        WHERE ur.user_id = ? AND ur.is_active = 1
          AND (ur.expires_at IS NULL OR ur.expires_at > NOW())`,
      [payload.id]
    );

    req.user = {
      ...rows[0],
      assignedRoles: roleRows.map(r => r.name),
      jamiats: parseIds(roleRows.flatMap(r => r.jamiat_ids)),
      jamaats: parseIds(roleRows.flatMap(r => r.jamaat_ids)),
    };
    next();
  } catch (err) {
    res.status(401).json({ error: 'Invalid token' });
  }
};

Token issuance on login: jwt.sign({ id, role }, JWT_SECRET, { expiresIn: JWT_EXPIRES_IN || '24h' }). The token carries only minimal claims; full profile is fetched on each request to ensure deactivation takes effect immediately.

Password hashing paradigm: bcrypt.hash(password, 10) — cost factor 10 (~100ms hash time, sufficient for brute-force resistance). Legacy plain-text passwords in imported user data are detected and re-hashed on first successful login.

5.2 Organizational Scoping Engine

The scoping engine is the most critical security component. It ensures users can only read and modify data within their organizational boundaries. 

// Simplified scope filter injection — applyScopeFilter()
// Called before route handlers on all case/applicant list endpoints

const BYPASS_ROLES = ['super_admin', 'admin'];
const APPROVE_SCOPE = 'jamiat';   // welfare, ZI, executive → jamiat-level scope
const FILL_SCOPE    = 'jamaat';   // counselors → jamaat-level scope

applyScopeFilter = (req, res, next) => {
  const { assignedRoles, jamiats, jamaats } = req.user;

  if (assignedRoles.some(r => BYPASS_ROLES.includes(r))) {
    req.scopeFilter = '';  // no WHERE restriction
    return next();
  }

  if (assignedRoles.some(r => APPROVE_ROLES.includes(r)) && jamiats.length) {
    req.scopeFilter = `AND c.jamiat_id IN (${jamiats.join(',')})`;
  } else if (assignedRoles.some(r => FILL_ROLES.includes(r)) && jamaats.length) {
    req.scopeFilter = `AND c.jamaat_id IN (${jamaats.join(',')})`;
  } else {
    // Counselor assigned directly to cases — no blanket scope, case-level check
    req.scopeFilter = `AND c.assigned_counselor_id = ${req.user.id}`;
  }
  next();
};
Security constraint: The req.scopeFilter fragment is only ever built from whitelisted integer arrays (parsed with parseInt() validation) — never from raw user input strings. SQL injection via scope manipulation is structurally impossible.

5.3 Case Identification Pipeline

Case identification is the entry point for all welfare cases. The pipeline involves ITS fetch, duplicate detection, income assessment, and conditional auto-case-creation on eligibility approval. 

POST /api/case-identifications
  │
  ├─ 1. Joi: validate its_number(8 chars), income fields, eligible_in
  ├─ 2. Duplicate check:
  │      SELECT id, status FROM case_identifications
  │       WHERE its_number = ? AND status IN ('pending','eligible')
  │       LIMIT 1
  │      → 409 if found
  │
  ├─ 3. ITS auto-fetch (non-blocking attempt):
  │      GET external_ITS_API/contact/{its_number}
  │      → merge full_name, phone, email, photo into insert payload
  │      → catch: proceed with manually entered data only
  │

5.4 Case Management & Status Machine

Cases transition through a defined set of statuses driven by workflow actions. The status machine is enforced server-side — clients cannot set arbitrary status values directly.

Action Endpoint From Status To Status Side Effects
POST /cases — draft status_history INSERT, notify DCM
PUT /cases/:id/welfare-approve welfare_review zi_review status_history, notify ZI users
PUT /cases/:id/welfare-reject welfare_review welfare_rework status_history, comment required, notify DCM
PUT /cases/:id/welfare-forward-rework welfare_review dcm_rework status_history, notify DCM
PUT /cases/:id/resubmit-welfare dcm_rework / welfare_rework welfare_review status_history, notify welfare
PUT /cases/:id/zi-approve zi_review next stage key Resolves next_stage_id from workflow_stages
PUT /cases/:id/zi-reject zi_review welfare_review comment required, notify welfare
PUT /cases/:id/executive-approve executive_L{N} executive_L{N+1} or manzoori Reads current_executive_level, increments or finalizes
PUT /cases/:id/executive-rework executive_L{N} welfare_review Resets executive level; comment required
POST /cases/:id/close any closed case_closures + case_closure_documents insert

All workflow action handlers share a common pattern:

// Shared workflow action pattern (simplified)
async function performWorkflowAction(caseId, toStatus, toStageId, actorId, comments) {
  const conn = await pool.getConnection();
  try {
    await conn.beginTransaction();

    // 1. Update case status + stage atomically
    await conn.execute(
      `UPDATE cases SET status=?, current_workflow_stage_id=?,
        current_stage_entered_at=NOW(),
        workflow_history=JSON_ARRAY_APPEND(workflow_history,'$',JSON_OBJECT(
          'stage',?,'status',?,'actor',?,'ts',NOW()
        ))
        WHERE id=?`,
      [toStatus, toStageId, toStageId, toStatus, actorId, caseId]
    );

    // 2. Append to status_history (append-only audit)
    await conn.execute(
      `INSERT INTO status_history (case_id, from_status, to_status, changed_by, comments)
         VALUES (?,?,?,?,?)`,
      [caseId, currentStatus, toStatus, actorId, comments]
    );

    await conn.commit();

    // 3. Async side effects (non-blocking)
    notifyNextStageActors(caseId, toStageId);
  } catch (e) {
    await conn.rollback();
    throw e;
  } finally { conn.release(); }
}
Transaction pattern: Status updates + audit log inserts always run inside a single DB transaction. A partial write (status updated but history not written) is impossible.

5.5 Counseling Form Architecture

The counseling form is the most complex data entry surface in the system. It uses an aggregate root pattern with lazy child record creation. 

GET /api/counseling-forms/case/:caseId
  │
  ├─ 1. SELECT counseling_forms WHERE case_id = ? (with all FK IDs)
  ├─ 2. IF NOT EXISTS → INSERT counseling_forms (case_id, is_complete=0) auto-create shell
  ├─ 3. For each section FK that is NOT NULL → JOIN fetch that section's data
  ├─ 4. For each section FK that IS NULL → return empty object for that section
  ├─ 5. Attach stage_permissions for req.user.role (from counseling_form_stage_permissions)
  └─ 6. Return merged form object + stage_permissions

PUT /api/counseling-forms/:formId/section/:section
  │
  ├─ section ∈ { personal_details, family_details, assessment,
  │               financial_assistance, economic_growth, declaration, attachments }
  │
  ├─ Route to section handler (e.g. updateFamilyDetails()):
  │
  │  family_details example:
  │  ├─ Joi validate section payload
  │  ├─ conn.beginTransaction()
  │  ├─ UPSERT family_details (INSERT ... ON DUPLICATE KEY UPDATE ...)
  │  ├─ DELETE family_members WHERE family_details_id = ?   -- full replace pattern
  │  │    then bulk INSERT new family_members rows
  │  ├─ UPDATE counseling_forms SET family_details_id = ? WHERE id = ?
  │  ├─ conn.commit()
  │  └─ 200 { message, section, updated: true }
  │
  └─ financial_assistance example (complex sub-tables):
     ├─ UPSERT financial_assistance core row
     ├─ DELETE + bulk INSERT: timeline, timeline_assistance, action_plan, mentors, qh_schedule
     └─ UPDATE counseling_forms FK
Full-replace pattern for sub-arrays: DELETE all child rows for the parent, then bulk INSERT the submitted array. Simpler than differential merge; safe because the entire section is submitted together. Transaction wraps both operations atomically.

5.6 Workflow & Approval Engine

Workflow stages are data-driven — stored in workflow_stages and resolved at runtime. No hardcoded stage sequences in application code. 

Stage resolution algorithm (PUT /cases/:id/workflow-action — generic):

1. FETCH current stage: workflow_stages WHERE id = case.current_workflow_stage_id
2. VALIDATE actor permissions:
   a. CHECK workflow_stage_users WHERE workflow_stage_id=? AND user_id=req.user.id
   b. OR CHECK workflow_stage_roles WHERE workflow_stage_id=? AND role_id IN (user's roles)
   → action='approve' requires can_approve=1; action='reject' requires can_reject=1
3. IF action='approve':
   a. IF stage.auto_advance_on_approve=1:
      → next_stage_id = stage.next_stage_id
      → resolve next stage's associated_statuses[0] as new status
   b. ELSE → use explicitly passed next_stage_id from request
4. Validate: stage.requires_comments_on_reject → 400 if empty on reject
5. performWorkflowAction(caseId, newStatus, newStageId, actorId, comments)
6. notifyStageActors(newStageId, caseId)  -- async

Executive level resolution (special case):
  current_executive_level = case.current_executive_level
  → IF next level exists in executive_levels (ordered by level_number):
       advance to executive_L{N+1} stage
     ELSE:
       advance to manzoori stage (final sanction)
  → UPDATE cases.current_executive_level = nextLevel || NULL

5.7 SLA Calculation Engine

SLA is calculated dynamically — not stored as a computed field — to always reflect current configuration changes. 

// utils/slaCalculator.js
function calculateSlaStatus(stageEnteredAt, slaValue, slaUnit,
                              slaWarningValue, slaWarningUnit) {
  const now = new Date();
  const entered = new Date(stageEnteredAt);

  const elapsedMs = now - entered;
  const breachMs  = toMs(slaValue, slaUnit);
  const warnMs    = breachMs - toMs(slaWarningValue, slaWarningUnit);

  if (elapsedMs >= breachMs)  return { status: 'breached', breachedAt: new Date(entered.getTime()+breachMs) };
  if (elapsedMs >= warnMs)    return { status: 'at_risk'  };
  return { status: 'on_time' };
}

// toMs() handles: hours, days, business_days (excludes weekends), weeks, months
// Business days calculation iterates calendar days, skipping Sat/Sun

SLA info is appended to every case in the list response as slaInfo: { status, breachedAt? }. The notification service's POST /notifications/check-sla endpoint iterates all open cases, recalculates SLA, and creates warning/breach notifications in bulk.

5.10 ITS External API Integration

The ITS (Institutional Information System) is an external API providing applicant identity data. Integration is designed to fail gracefully. 

GET /api/applicants/fetch-from-api/:itsNumber

const fetchFromIts = async (itsNumber) => {
  try {
    const res = await axios.get(`${ITS_API_BASE}/contact/${itsNumber}`, {
      headers: { Authorization: `Bearer ${ITS_API_KEY}` },
      timeout: 5000  // 5s max; non-blocking for UI
    });
    return { success: true, data: normalizeItsResponse(res.data) };
  } catch (err) {
    // Log for monitoring; surface 'not_found' vs 'api_error' to caller
    if (err.response?.status === 404)
      return { success: false, reason: 'not_found' };
    return { success: false, reason: 'api_error', message: err.message };
  }
};

5.11 Notification Service

// services/notificationService.js
const createNotification = async (userIds, caseId, title, message, type='info') => {
  if (!userIds?.length) return;
  const placeholders = userIds.map(() => '(?,?,?,?,?,0)').join(',');
  const values = userIds.flatMap(id => [id, caseId, title, message, type]);
  await pool.execute(
    `INSERT INTO notifications (user_id,case_id,title,message,type,is_read) VALUES ${placeholders}`,
    values
  );
  // Bulk INSERT — one query for N recipients (prevents N+1 for stage with 10+ users)
};

// SLA check (POST /api/notifications/check-sla)
async function runSlaCheck() {
  const openCases = await fetchOpenCasesWithStageInfo();
  for (const c of openCases) {
    const sla = calculateSlaStatus(c.stage_entered_at, c.sla_value, c.sla_unit,
                                     c.sla_warning_value, c.sla_warning_unit);
    if (sla.status === 'at_risk' || sla.status === 'breached') {
      const actors = await getStageActorIds(c.current_workflow_stage_id);
      await createNotification(actors, c.id,
        `SLA ${sla.status} — Case ${c.case_number}`,
        `Case has been in "${c.stage_name}" for ${c.elapsed} ${c.sla_unit}.`,
        sla.status === 'breached' ? 'error' : 'warning'
      );
    }
  }
}

9. Performance & Scalability

9.1 Database Query Optimisation

  • Composite indexes on all high-cardinality filter combinations (status + jamiat_id, case_id + quarter + year).
  • SELECT specificity: All queries name explicit columns. No SELECT * in production code — avoids fetching 100KB photo blobs in list queries where only IDs and names are needed.
  • Pagination enforcement: All list endpoints require LIMIT/OFFSET. Maximum limit=500 hard cap. No unbounded result sets.
  • N+1 prevention: Notification bulk INSERT; case list uses JOINs not per-row sub-selects; family members fetched in one WHERE family_details_id IN (?).
  • Connection pooling: mysql2 createPool({ connectionLimit: 10 }) handles burst concurrency without connection overhead per request.

9.2 Frontend Performance

  • React Query caching: Case list cached for 5 minutes by default; stale-while-revalidate prevents empty loading states on navigation.
  • Code splitting: React Router lazy-loads page components — initial bundle does not include PDF viewer, fund utilization form builder, or role management pages.
  • Photo handling: Applicant/user photos stored as base64 in DB. List queries join only id/name — photos fetched only on detail view to prevent bloated list responses.
  • Excel import chunking: Server-side xlsx parsing uses streaming mode (readFile with row-by-row processing) — prevents memory spikes for 500+ row imports.

9.3 Scaling Ceiling & Upgrade Path

Bottleneck Current Limit Upgrade Path
Concurrent users ~50 (single Node.js process) PM2 cluster mode (N = CPU cores); Nginx upstream load balancing
File storage Server disk (EBS volume) AWS S3 + signed URL downloads; Multer S3 storage engine
PDF generation ~3s/doc (Puppeteer) Worker thread pool; queue-based generation (Bull/BullMQ)

  }]

11. Error Handling & Logging

11.1 Global Error Handler

// backend/index.js — last middleware registered
app.use((err, req, res, next) => {
  const requestId = req.headers['x-request-id'] || uuidv4();

  // Structured log: never expose stack trace to client
  console.error(JSON.stringify({
    level: 'error',
    requestId,
    method: req.method, path: req.path,
    userId: req.user?.id,
    error: err.message,
    stack: err.stack,
    ts: new Date().toISOString()
  }));

  // Multer-specific error handling
  if (err.code === 'LIMIT_FILE_SIZE')
    return res.status(400).json({ error: 'File too large. Maximum 10MB per file.' });
  if (err.code === 'LIMIT_FILE_COUNT')
    return res.status(400).json({ error: 'Too many files. Maximum 10 per upload.' });

  // DB constraint violations
  if (err.code === 'ER_DUP_ENTRY')
    return res.status(409).json({ error: 'Duplicate entry — record already exists.' });

  // Safe generic fallback
  res.status(500).json({ error: 'Internal server error', requestId });
});

11.2 Validation Error Pattern (Frontend)

// Axios interceptor — frontend/src/utils/api.js
api.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      authContext.logout();        // clear token, redirect to login
    }
    if (error.response?.status === 403) {
      toast.error('Access denied. Insufficient permissions.');
    }
    return Promise.reject(error);
  }
);

11.3 Log Structure

Log Level Events Destination
ERROR Unhandled exceptions, DB errors, external API failures, file write failures PM2 error log + /var/log/pm2/facts-error.log
WARN JWT near-expiry, SLA breaches, ITS API fallback triggered PM2 out log
INFO Server start, DB connection pool ready, migration applied PM2 out log
DEBUG Query plans, scope filters applied (NODE_ENV=development only) Console (not in production)
No sensitive data in logs: Passwords, JWT tokens, ITS API keys, and full file paths are never logged. User IDs and case numbers are included for traceability. All log output is structured JSON for parsing by log aggregators (e.g. CloudWatch, Datadog).
