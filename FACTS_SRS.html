1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document provides a complete and detailed description of the requirements for the FACTS — File and Case Tracking System. It is intended for:

  • Developers — to understand technical requirements and implement features correctly
  • Testers — to create test cases and validate system behavior against acceptance criteria
  • Project Managers — to track scope, prioritize backlog, and monitor delivery progress
  • Stakeholders — to understand system capabilities and approve functional specifications
  • Maintenance Team — to reference current specifications during updates and troubleshooting

This document describes the software's functional requirements, external interfaces, design constraints, performance attributes, and all non-functional requirements, organized per the IEEE 830-1998 standard.

1.2 Scope

FACTS is a comprehensive, full-stack web application that digitalizes and centralizes the complete lifecycle of welfare case management for the Baaseteen and SHND organizations. The system manages:

  1. User account and role management — JWT authentication, granular RBAC, Jamiat/Jamaat scope isolation, multi-role user support
  2. Case identification and eligibility screening — pre-assessment intake with ITS API integration, income assessment, and duplicate detection
  3. Case lifecycle management — creation, assignment, status tracking, SLA monitoring, audit trail, and closure
  4. Digital counseling forms — 8-section wizard covering Personal Details, Family Details, Assessment, Financial Assistance, Economic Growth, Declaration, Attachments, and Manzoori
  5. Multi-level workflow approvals — configurable per case type, supporting up to 10 stages and 5 executive approval tiers, with approve/reject/rework flows
  6. Cover letter generation — automated PDF generation from consolidated case data, with welfare department digital signature capture
  7. Payment and financial management — Qardan Hasana and Enayat disbursement schedules, installment tracking, and repayment confirmation
  8. Fund utilization reporting — quarterly reassessment forms with configurable admin-defined question schema, photo uploads, and PDF export
  9. Welfare checklist — configurable checklist with drag-and-drop categories and per-case completion enforcement
  10. Document management — multi-file attachment support per case and counseling section, with type and size validation
  11. Notification system — in-app notifications for case assignments, status changes, and SLA alerts
  12. Dashboard analytics — role-scoped statistics, pipeline overviews, and recent activity feeds
  13. Reporting — Excel import/export for applicants and users; PDF downloads for cover letters and quarterly reports
  14. Master data management — configuration of Jamiat, Jamaat, case types, workflow stages, executive levels, relations, education levels, and occupations

Out of Scope: Mobile native apps, SMS notifications, video conferencing, payment gateway integration, public applicant self-registration portal, multi-tenant SaaS architecture, third-party CRM integration.

3. System Features and Functional Requirements

3.1 Authentication and Session Management

3.1.1 Description

The authentication module provides stateless JWT-based login, session management, and role enforcement for all system users. RBAC is enforced at the API route level using a resource:action permission model. All data queries are additionally filtered by Jamiat/Jamaat scope at the database layer.

3.1.2 Functional Requirements

SRS-AUTH-001: JWT Login

Attribute Detail
Priority Critical
Access Public (unauthenticated)
Input username (string, required) or email (string, required); password (string, min 8 chars, required)
Processing 1. Validate input fields; 2. Lookup user by username or email in users table; 3. Check is_active = 1; reject inactive accounts with 403; 4. Compare password against password_hash using bcrypt.compare(); 5. On match: generate signed JWT containing userId, username, role, jamiat_id, jamaat_id, executive_level; token valid for 24 hours; 6. On failure: return HTTP 401 without revealing whether username or password was incorrect
Output HTTP 200: { token, user: { id, username, role, full_name, permissions[] } }; HTTP 401 on failure
Rate Limiting Maximum 5 failed attempts per IP per 15 minutes via express-rate-limit
Database SELECT from users WHERE username = ? OR email = ?

SRS-AUTH-002: JWT Verification Middleware

Attribute Detail
Priority Critical
Trigger Every request to a protected API endpoint
Processing 1. Extract Bearer token from Authorization header; 2. Verify JWT signature using JWT_SECRET; 3. Check token expiry; 4. Re-fetch user from DB to validate is_active status; 5. Attach decoded user object to req.user; 6. Reject with HTTP 401 if any check fails
Output Request proceeds with req.user populated; HTTP 401 if unauthorized

SRS-AUTH-003: RBAC Permission Middleware

Attribute Detail
Priority Critical
Trigger After JWT verification, on every protected endpoint that specifies a required permission
Processing 1. Resolve user's effective permissions from all assigned roles in user_roles junction table; 2. Check if required resource:action string exists in the resolved permission set; 3. Super Admin bypasses all permission checks; 4. Reject with HTTP 403 if permission not present
Permission Format resource:action — e.g., cases:read, cases:create, counseling_forms:update, reports:export
Output Request proceeds if permitted; HTTP 403 Forbidden if not

SRS-AUTH-004: Jamiat/Jamaat Scope Filter

Attribute Detail
Priority Critical
Trigger Any list/search query for cases, applicants, or users
Processing 1. Inspect req.user.role and req.user.jamiat_id; 2. If role is Super Admin or Admin: no WHERE clause injected; 3. If role is DCM or Welfare Reviewer: inject WHERE cases.jamiat_id = req.user.jamiat_id; 4. If role is Counselor: inject WHERE cases.jamaat_id = req.user.jamaat_id; 5. Scope constraint applied at SQL query level — not just frontend filtering
Output Result set filtered to authorized organizational scope only

SRS-AUTH-005: Profile Retrieval and Permission Refresh

Attribute Detail
Priority High
Endpoint GET /api/auth/profile
Processing 1. Validate JWT token; 2. Fetch full user profile from DB including all assigned roles and per-role Jamiat/Jamaat assignments; 3. Resolve and return complete permission tree to React frontend for UI rendering decisions
Output Full user profile JSON with assigned_roles[] and flattened permissions[]

SRS-AUTH-006: Logout

Attribute Detail
Priority High
Processing JWT logout is client-side token discard (stateless architecture). Server provides POST /api/auth/logout endpoint for audit logging purposes. No server-side session state is maintained.

3.2 User Management

3.2.1 Description

The user management module allows administrators to create, view, update, deactivate, and export user accounts across all 8 roles. It supports individual and bulk operations including Excel import with ITS API enrichment.

3.2.2 Functional Requirements

SRS-USER-001: Create User

Attribute Detail
Priority Critical
Access Admin, Super Admin
Input full_name (required), username (required, unique), email (required, unique), role (required), password (optional — defaults to TempPassword123!), phone, its_number, photo (base64), jamiat_id, jamaat_id, executive_level, role_jamiat_jamaat[]
Processing 1. Validate all required fields; 2. Enforce username and email uniqueness across users table; 3. Hash password using bcryptjs (10 rounds); 4. INSERT into users; 5. If role_jamiat_jamaat[] provided, INSERT per-role scope into user_roles; 6. Log creation in audit trail
Output HTTP 201: { message, userId }; HTTP 409 on duplicate username/email

SRS-USER-002: Update User

Attribute Detail
Priority High
Access Admin, Super Admin
Input user_id, any updatable field; password change is optional (re-hashed if provided)
Processing 1. Validate fields; 2. Enforce uniqueness on username/email excluding current user; 3. UPDATE users; 4. Replace user_roles entries if roles/scope provided; 5. Log update in audit trail
Output HTTP 200: { message: "User updated successfully" }

SRS-USER-003: Activate / Deactivate User

Attribute Detail
Priority High
Access Admin, Super Admin
Processing PATCH users SET is_active = 0 or 1; deactivated users rejected at JWT verification middleware mid-session; account data preserved for audit trail

SRS-USER-004: List and Filter Users

Attribute Detail
Priority High
Access Admin, Super Admin
Input Query params: page, limit (10/20/50/100/500), role, is_active, search (name/email/username), jamiat_id
Processing SELECT from users JOIN user_roles with pagination and filter WHERE clauses; return total count and paginated data
Output { total, page, limit, data: [ { id, username, role, assigned_roles[], is_active } ] }

SRS-USER-005: Bulk Import via Excel

Attribute Detail
Priority Medium
Access Admin, Super Admin
Input Excel file (.xlsx) matching downloadable template from GET /api/users/export/template
Processing 1. Parse file using ExcelJS; 2. Validate each row; 3. Check username/email uniqueness; 4. INSERT valid rows with hashed passwords; 5. Optionally call ITS API for users with ITS numbers to enrich profiles; 6. Return row-level results: inserted/updated/skipped/errors
Output Import summary with success count and per-row error details

SRS-USER-006: Export Users to Excel

Attribute Detail
Priority Medium

3.3 Case Identification and Eligibility Screening

3.3.1 Description

The case identification module is the intake gateway for all welfare applicants. It integrates with the external ITS API for automated demographic prefill, enforces duplicate detection, captures income assessment fields, and governs the transition to formal case creation upon an eligible decision.

3.3.2 Functional Requirements

SRS-ID-001: Create Case Identification

Attribute Detail
Priority Critical
Access DCM, Admin, Super Admin
Input its_number (8-digit, required), eligible_in (case_type_id, required), total_family_members (0–25), earning_family_members (0–20), individual_income (0–9,999,999), family_income (0–9,999,999), remarks (optional), Jamiat/Jamaat assignment
Processing 1. Validate ITS number format (8 numeric digits); 2. Query applicants table — if found, link; else trigger ITS API lookup and create applicant record; 3. Check for active duplicate case on same ITS number (block if active case exists in non-closed/non-rejected state with HTTP 409); 4. INSERT into case_identifications with status = 'pending'; 5. Log income baseline
Output HTTP 201: { id, its_number, status: "pending", applicant: {...} }
Database INSERT case_identifications; INSERT applicants if new

SRS-ID-002: ITS Number Lookup (Auto-Fetch)

Attribute Detail
Priority Critical
Endpoint GET /api/applicants/lookup/:itsNumber
Processing 1. Validate 8-digit numeric format; 2. Query local applicants table first (cache-first strategy); 3. If not found locally, proxy request to external ITS API with masked credentials in Node server; 4. Parse response capturing: full_name, age, gender, phone, email, photo (base64), address, jamiat, jamaat; 5. Cache result in applicants table; 6. Return formatted payload to frontend for auto-population
Error Handling ITS API timeout (5 seconds): return graceful empty response; network error: log and return HTTP 503 with user-friendly message
Output HTTP 200: populated applicant object; HTTP 404: applicant not found

SRS-ID-003: Review — Eligible or Ineligible

Attribute Detail
Priority Critical
Access Permission: case_identification:approve
Input identification_id, decision ('eligible' or 'ineligible'), review_remarks (required on ineligible)
Processing Eligible path: 1. UPDATE case_identifications SET status = 'eligible'; 2. Auto-generate case number (e.g., BS-0001, sequential per case type); 3. INSERT into cases table with status = 'Draft', linked to applicant; 4. Auto-create counseling_forms record linked to new case; 5. Send notification. Ineligible path: 1. UPDATE status = 'ineligible'; 2. Store review_remarks, reviewed_by, reviewed_at; 3. Send notification
Output HTTP 200: { message, caseId, case_number } (eligible); HTTP 200: { message } (ineligible)

SRS-ID-004: Edit Assessment Fields with Income Log

Attribute Detail
Priority High
Access Permission: case_identification:edit
Input individual_income, family_income, eligible_in, total_family_members, earning_family_members, remarks
Processing 1. UPDATE case_identifications; 2. For each income field change: INSERT into case_identification_income_logs (identification_id, field_name, old_value, new_value, changed_by, changed_at); 3. Full audit trail maintained for all income modifications
Output HTTP 200: { message: "Case identification updated" }

SRS-ID-005: Revert Ineligible to Pending

Attribute Detail
Priority Medium
Access Permission: case_identification:edit
Processing 1. Validate current status is 'ineligible'; 2. UPDATE status = 'pending'; 3. Clear reviewed_by, reviewed_at, review_remarks; 4. Log revert action in audit trail

3.4 Case Management and Lifecycle Tracking

3.4.1 Description

The core case management module handles the full lifecycle of each welfare case from creation through closure. It manages assignments, workflow stage transitions, comments, SLA tracking, and maintains a complete status history audit trail.

3.4.2 Functional Requirements

SRS-CASE-001: Create Case

Attribute Detail
Priority Critical
Access Admin, DCM (auto-triggered from eligible case identification)
Input applicant_id (or inline applicant_data), case_type_id, optional: assigned_dcm_id, assigned_counselor_id, jamiat_id, jamaat_id
Processing 1. Generate case number (sequential per case type, zero-padded — e.g., BS-0001); 2. INSERT into cases with status = first workflow stage for case type; 3. Auto-create linked counseling_forms record; 4. Record stage entry timestamp in cases.current_stage_entered_at; 5. Notify assigned counselor
Output HTTP 201: { message, caseId, case_number }
Database INSERT cases, INSERT counseling_forms, INSERT case_status_history

SRS-CASE-002: Assign Case

Attribute Detail
Priority Critical
Access Admin, DCM
Input case_id, counselor_id (filtered to Counselor role within Jamaat scope)
Processing 1. Validate counselor has Counselor role in matching Jamaat; 2. UPDATE cases.assigned_counselor_id; 3. Create in-app notification for assigned counselor; 4. Log assignment action in workflow_logs
Output HTTP 200: { message: "Case assigned successfully" }

SRS-CASE-003: Case Comments

Attribute Detail
Priority High
Access All roles with cases:read permission
Input case_id, comment_text (required), comment_type (general / rejection / rework)

SRS-CASE-004: SLA Tracking

Attribute Detail

7. System Architecture

7.1 Architectural Pattern

FACTS follows a decoupled three-tier architecture that cleanly separates client presentation, business logic, and data persistence into independently maintainable layers:

  • Presentation Tier — React 18.2 Single-Page Application compiled with Vite, served as static build artifacts by Nginx. All UI rendering is client-side; server only provides JSON data.
  • Application Tier — Node.js v16+ with Express.js handling RESTful API requests, business rule enforcement, workflow progression, PDF generation, and external service integration.
  • Data Tier — MySQL 8.0 with InnoDB for relational transactional storage, plus local filesystem for binary file attachments and generated PDFs.

Architectural advantages: Clean API contract between frontend and backend; frontend and backend can be deployed and scaled independently; technology boundaries are explicit; testability is high due to stateless API design.

7.3 Request Lifecycle

Step 1: React SPA makes Axios HTTP request with Authorization: Bearer <JWT>
        │
Step 2: Express receives request
        → Helmet adds security headers (CSP, X-Frame-Options, HSTS)
        → CORS validates origin against FRONTEND_URL env var
        → Rate limiter checks request frequency (login: 5/15min; general: configurable)
        │
Step 3: JWT Auth Middleware:
        - Decodes and verifies JWT signature using JWT_SECRET
        - Re-queries DB to confirm user.is_active = 1
        - Attaches user object to req.user
        - Returns 401 if token invalid, expired, or user deactivated
        │
Step 4: RBAC Permission Middleware:
        - Checks req.user.role against required resource:action permission
        - For scoped roles (DCM, Counselor): injects WHERE jamiat_id / jamaat_id
          constraints into query context — non-admin users cannot bypass
        - Returns 403 if insufficient permission
        │
Step 5: Joi Validation:
        - Validates request body and query params against defined schema
        - Returns 400 with field-level error details on validation failure
        │
Step 6: Route Handler (Business Logic):
        - Executes parameterized MySQL queries via connection pool
        - Calls services: emailService, notificationService, slaCalculator
        - Invokes PDF generators (PDFKit / Puppeteer) as needed
        - Handles file I/O via Multer streams
        │
Step 7: Response:
        - Returns JSON payload with standard HTTP status code
        - React Query caches response; UI state updates reactively
        - Toast notification displayed to user

8.3 SLA Calculation Flow

Case advances to new workflow stage
       │
       ├── Record current_stage_entered_at = NOW()
       │   Reset sla_status = "on_time"
       │
SLA check triggered (POST /api/notifications/check-sla or scheduled event)
       │
       ├── For each active case (non-terminal status):
       │       elapsed = NOW() - current_stage_entered_at
       │       stage.sla_value + stage.sla_unit → convert to hours
       │       stage.sla_warning_value + sla_warning_unit → convert
       │
       ├── If elapsed ≥ sla_value (breach):
       │       Set sla_status = "breached"
       │       Record sla_breached_at timestamp
       │       Notify: assigned_counselor + relevant approver + admin
       │       Notification type: "error"
       │
       ├── If elapsed ≥ sla_warning_value but < sla_value (at risk):
       │       Set sla_status = "at_risk"
       │       Notify: assigned_counselor + relevant approver
       │       Notification type: "warning"
       │
       └── Else:
               sla_status = "on_time"  (returned on case list API responses)

8.4 ITS Auto-Fetch Data Flow

User types ITS number in Case Identification / Applicant form
       │
       ├── Frontend: GET /api/applicants/fetch-from-api/:itsNumber
       │
       ├── Backend: Query local applicants table by its_number
       │       If found → return record with source: "database"
       │       (No external call needed — cached locally)
       │
       ├── If not found locally → proxy to external ITS API endpoint
       │       Timeout limit: 5 seconds (graceful fallback on failure)
       │       Parse JSON response → map to applicant schema fields:
       │         full_name, age, gender, phone, email, photo (base64),
       │         jamiat_name, jamaat_name, address
       │       Return mapped object with source: "api"
       │
       ├── Frontend auto-populates form fields from returned object
       │       Editable fields remain open for user corrections
       │
       └── User reviews → saves applicant record to local DB
