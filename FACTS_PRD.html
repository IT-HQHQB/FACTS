1. Product Vision & Overview

1.1 Vision Statement

FACTS aims to be the definitive digital platform for welfare case management — eliminating fragmented paper-based processes, enforcing consistent multi-level approval workflows, and providing every stakeholder with real-time visibility from the moment a case is identified through to financial disbursement and fund utilization tracking.

1.2 Product Summary

FACTS is a full-stack web application built for Baaseteen and SHND organizations. It digitizes the entire welfare case lifecycle: from eligibility screening and applicant profiling through counseling, multi-authority approvals, financial disbursement scheduling, and ongoing quarterly reassessment. The system enforces role-scoped access, automated notifications, SLA tracking, and comprehensive audit trails at every step.

Key Capabilities at a Glance:

Capability What It Does
Case Identification Pre-assessment eligibility screening with ITS integration, income evaluation, and auto-case creation on approval
Case Management Full lifecycle tracking with configurable multi-stage approval workflows, comments, and audit trails
Applicant Profiles Rich applicant records auto-populated from the ITS API including photos, family details, and contact info
Counseling Forms Multi-section digital forms (7 sections) replacing paper-based counseling documentation
Cover Letters Automated PDF generation for case cover letters with submission workflows
Payment Schedules Yearly disbursement plans with monthly repayment tracking and confirmation
Fund Utilization Quarterly reassessment reports with configurable dynamic form builder and photo documentation
Welfare Checklist Configurable checklist categories and items with per-case completion tracking
RBAC Granular role-based access at resource + action level with Jamiat/Jamaat organizational scoping
SLA Tracking Per-stage SLA values with automated breach detection and in-app alerts
Notifications In-app alerts for status changes, assignments, and SLA warnings
Analytics Dashboard Role-specific dashboards with case pipeline views and activity feeds

2. Problem Statement

2.1 Current Challenges

Challenge Impact
Paper-based case intake Eligibility assessments were manual; no central record of who was screened, their income, or their outcome
Disconnected approval chain Cases moved between counselors, welfare reviewers, ZI, and executives via email and in-person; no trackable audit trail
No standardized counseling forms Each counselor used their own format; comparison across cases was impossible
Manual financial tracking Payment disbursements and repayment schedules were tracked in spreadsheets susceptible to errors and loss
No fund utilization visibility There was no systematic way to track how disbursed funds were used quarter-by-quarter
Organizational scope confusion Staff in one Jamiat could view or act on cases belonging to another, violating data boundaries
Duplicate case risk No enforcement preventing an applicant from having multiple concurrent active cases
Zero SLA enforcement Cases could sit at a stage indefinitely with no system-level warnings or escalation

2.2 Desired Outcome

A single platform where:

  • Field workers can screen applicants and submit eligibility assessments digitally with ITS auto-fill
  • Counselors can complete structured multi-section forms and attach supporting documents
  • Welfare reviewers, ZI officers, and executives can approve or return cases through a traceable workflow
  • Finance staff can manage payment schedules and confirm disbursements
  • Administrators can configure workflows, roles, master data, and monitor every case from a single dashboard
  • All users receive real-time notifications and see only the data relevant to their organizational scope

6. Feature Requirements

6.1 Authentication & Access Control

Purpose: Securely authenticate users and enforce role-based, organizationally-scoped access across the platform.

Feature Description Priority
JWT authentication Stateless JWT tokens (24-hour expiry) issued on login; verified on every request Critical
bcrypt password storage All passwords hashed with bcryptjs; minimum 6 characters enforced on change Critical
Multi-identifier login Users can authenticate with either username or email address High
Role-based access control Permissions defined at resource + action level (e.g., cases:create, case_identification:approve) Critical
Organizational scope enforcement Jamiat-scope users see only Jamiat-assigned cases; Jamaat-scope users see only Jamaat-assigned cases Critical
Multi-role assignment Users can hold multiple roles simultaneously; primary display role can be toggled High
Token user-status check Every authenticated request checks whether the user account is still active High
Rate limiting on login Brute-force protection on the login endpoint via express-rate-limit Medium

6.2 User & Role Management

Purpose: Allow administrators to create, configure, and manage all user accounts and permission roles.

Feature Description Priority
User CRUD Create/read/update/delete users with name, email, phone, ITS number, photo (base64), and role assignments Critical
Role CRUD Create/edit roles with display name, description, and granular permission sets Critical
Permission assignment 15 resources × multiple actions; assigned at role level and enforced by middleware Critical
Counseling form stage permissions Per-role read/update access configurable per counseling form stage High
Executive level assignment Assign users to numbered executive approval levels (1–5); Admin/Super Admin only High
Activate/deactivate users Toggle user status; deactivated users cannot authenticate High
Bulk import via Excel Import users from .xlsx/.xls with per-row validation and error reporting Medium
ITS API sync Fetch and update contact details from the external ITS system for users with ITS numbers Medium
Export to Excel Download user lists filtered by role or status Medium

6.3 Case Identification

Purpose: Pre-assess applicant eligibility before creating a full case, ensuring only eligible candidates enter the workflow.

Feature Description Priority
ITS auto-fill Enter ITS number → system fetches name, age, gender, photo, contact, Jamiat/Jamaat from DB or external API Critical
Duplicate ITS detection System checks if ITS number already has an existing case identification record and blocks re-entry Critical
Income assessment Record individual income, family income, total family members, earning members, and case type assignment Critical
Eligible / Ineligible review Authorized reviewers mark records as eligible (auto-creates a case) or ineligible with remarks Critical
Edit assessment fields Edit income fields and eligible_in on pending or ineligible records; changes logged to audit table High
Revert to pending Revert an ineligible record back to pending for re-review High
Income change audit log Every income field modification is recorded with old value, new value, changed_by, and timestamp High
Filter and search Filter by status, case type, Jamiat, Jamaat; search by ITS or full name with pagination High

6.4 Case Management

Purpose: Manage the full lifecycle of welfare cases from creation to closure through configurable multi-stage workflows.

Feature Description Priority
Case creation Create cases manually or auto-generate from eligible case identifications; assigns auto-incremented case number (e.g., BS-0001) Critical
Case assignment Assign DCM user and counselor per case; restrict visible cases by Jamiat/Jamaat scope Critical
Workflow actions Approve, reject, rework, resubmit — each action advances or returns the case to a defined stage Critical
Multi-level executive approval Cases route through up to 5 configurable executive levels; next level determined automatically Critical
Stage-specific comments Users add general or workflow-step comments; all comments time-stamped and stored per case High
Status history Every status transition is logged with from/to status, changed_by, and timestamp High
SLA tracking Each stage has configurable SLA value and warning threshold; system calculates on_time / at_risk / breached High
Case closure Close a case with a reason and 1–10 supporting documents (PDF, DOC, DOCX, XLS, XLSX, JPG, PNG, max 10 MB each) High
Duplicate case flag Cases are flagged if the same ITS number has multiple active cases Medium
Filter and search Filter by status, case type, assigned DCM, counselor, Jamiat, Jamaat, workflow stage, duplicate flag High

6.5 Applicant Management

Feature Description Priority

6.6 Counseling Forms

Purpose: Replace paper-based counseling documentation with a structured, section-by-section digital form linked to each case.

6.6.1 Form Sections

# Section Key Content
1 Personal Details ITS, name, age, education, Jamiat/Jamaat, contact, address, occupation
2 Family Details Family structure, wellbeing areas (food/housing/health/education/deeni), income & expense breakdown, assets & liabilities, family members table
3 Assessment Applicant background, proposed business details, counselor evaluation of demand/supply/competition
4 Financial Assistance Assistance required, self-funding, repayment schedule, QH fields, support mentors
5 Economic Growth 5-year revenue, expense, and profit projections; asset/liability trajectory
6 Declaration Applicant, counselor, and TR Committee digital signatures with date and confirmation text
7 Attachments Workplace photos, quotations, income tax returns, PAN card, Aadhar card, other documents

9.2 Architecture Pattern

FACTS follows a REST API + SPA architecture. The Express backend exposes 150+ API endpoints organized by resource. The React frontend consumes these endpoints via Axios with React Query managing server-state caching. Authentication is stateless via JWT. 

Browser (React SPA)
  ↕ HTTP/HTTPS (Axios + React Query)
Express API (Node.js)
  ↕ Parameterized Queries (mysql2)
MySQL Database (InnoDB)
  ↕ Multer
Filesystem (uploads/)
  ↕ External HTTP
ITS API (applicant data source)

9.3 Key Design Patterns

Pattern Usage
REST Resource Controllers 25 route files; each resource is a self-contained Express router with CRUD + actions
JWT Middleware Every protected route verifies token and loads user + permissions before handler runs
Scope Filtering Middleware injects Jamiat/Jamaat scope into queries based on user's assigned role
Parameterized Queries All DB queries use mysql2 parameterized binding — no string concatenation
Soft Deletes Workflow stages and roles use is_active flag; records are never hard-deleted
Junction Tables Many-to-many: user↔role, role↔workflow_stage, user↔workflow_stage
Audit Columns created_by, updated_by, created_at, updated_at on all core tables

11. Performance & Reliability

11.1 Performance Targets

Operation Target Time

Page load (React SPA) < 3 seconds
Complex JOIN query (cases list) < 2 seconds
PDF cover letter generation < 5 seconds
Quarterly report PDF generation < 10 seconds
Excel import (bulk users) < 15 seconds
ITS API fetch (single) < 3 seconds

11.2 Reliability Targets

Requirement Target
System uptime ≥ 99.5% (excluding planned maintenance)
Data integrity InnoDB foreign key constraints + parameterized queries on all operations
Error handling Structured JSON error responses; no stack traces exposed to clients in production
Audit trail completeness 100% of status changes, income edits, and approval actions logged

11.3 Optimization Strategies

  • MySQL connection pool (10 connections) via mysql2 to avoid connection overhead
  • React Query client-side caching with configurable stale times per resource
    • Pagination with LIMIT / OFFSET on all list endpoints
  • Specific column selection in queries; no SELECT * on production tables
  • ITS API results cached in the local DB to avoid repeated external calls
  • Multer file size limits enforced server-side before reaching business logic

12. Security & Compliance

12.1 Authentication Security

Measure Implementation
Password hashing bcryptjs — no plain-text passwords stored
JWT tokens Signed with strong secret; 24-hour expiry; verified on every request
User-status check Token verification also checks is_active = 1 — deactivated users are blocked immediately
Login rate limiting Configurable max attempts per IP window via express-rate-limit

12.2 Data Protection

Threat Protection
SQL Injection 100% parameterized queries via mysql2 — no string concatenation in SQL
XSS React's JSX escapes output by default; API responses are JSON only
CSRF Stateless JWT eliminates session cookie CSRF vectors
Unauthorized access RBAC middleware on every route; scope filtering injects Jamiat/Jamaat constraints into DB queries
Input validation Joi schema validation on all POST/PUT request bodies

12.3 File Security

Measure Implementation
File type validation Multer MIME-type + extension whitelist (PDF, DOC, DOCX, XLS, XLSX, JPG, PNG)
File size limit 10 MB per file enforced by Multer before file is written to disk
Isolated storage Files stored in uploads/ directory; never served directly — only via authenticated download endpoint

12.4 Infrastructure Security

Measure Implementation
HTTP security headers Helmet middleware sets Content-Security-Policy, X-Frame-Options, etc.
CORS Restricted to configured frontend origin only
Environment secrets DB credentials and JWT secret in .env; never committed to version control
HTTPS Required in production via Nginx SSL termination

14. Assumptions & Constraints

14.1 Assumptions

ID Assumption
A-1 Users have access to modern web browsers (Chrome, Firefox, Safari, Edge)
A-2 The external ITS API is accessible from the server network at runtime
A-3 Each applicant has a unique ITS number that serves as their primary identifier
A-4 Organizational hierarchy follows a two-level structure: Jamiat (parent) → Jamaat (child)
A-5 Assessment scoring for Qardan Hasana follows fixed repayment structures
A-6 Concurrent users will not exceed 100 at any time in the current deployment
A-7 File storage will remain within server disk capacity; S3 migration planned for scale

14.2 Constraints

ID Constraint
C-1 Technology stack is Node.js + React + MySQL — no framework migration planned
C-2 File storage is local filesystem only; no cloud object storage currently integrated
C-3 PDF generation requires PDFKit or Puppeteer; Puppeteer needs Chromium on the server
C-4 ITS API is an external dependency; outages degrade auto-fill functionality but do not block system use
C-5 Single-server deployment — no load balancing or horizontal scaling currently
C-6 Email notifications require SMTP configuration; optional and disabled by default
C-7 No real-time WebSocket support; notifications require page refresh or manual polling

15. Risks & Mitigation

ID Risk Likelihood Impact Mitigation Strategy
R-1 Data loss / DB corruption Low Critical Automated DB backups; InnoDB foreign keys; parameterized queries throughout
R-2 ITS API downtime Medium Medium Graceful fallback to local DB; manual data entry remains available
R-3 Unauthorized cross-Jamiat data access Low High Scope filtering enforced in middleware on every query; tested per role
R-4 Duplicate active cases per applicant Medium High Server-side one-active-case-per-ITS enforcement before case creation
R-5 File upload abuse Medium Medium Multer type/size validation; files stored outside web root; served via authenticated endpoint
R-6 Performance degradation at scale Medium Medium DB indexing on all FKs and search fields; pagination; React Query caching
R-7 Puppeteer PDF failures Low Medium PDFKit as alternative generator; error handling returns clear failure message
R-8 SLA misconfiguration Medium Low Admin UI for SLA editing; SLA calculation isolated in slaCalculator.js utility
