1. Executive Summary

1.1 Purpose

This Business Requirements Document defines the complete functional and non-functional requirements for the FACTS Case Management System — a comprehensive full-stack web application designed to manage the entire lifecycle of welfare cases for the Baaseteen and SHND (Social Help and Development) organizations. FACTS digitizes and centralizes every stage of welfare case operations, from applicant identification and eligibility assessment through multi-level approval workflows, financial disbursement, and quarterly fund utilization tracking.

1.2 Background

Prior to FACTS, administrative teams at Baaseteen and SHND relied on deeply decentralized, manual welfare review mechanisms. Paper-based processes led to prolonged service-level turnaround times, opaque executive approval chains, delays in financial disbursements to beneficiaries, and unreliable quarterly performance assessments. The absence of a unified tracking platform made audit trails incomplete and reporting cumbersome. FACTS was commissioned to replace these manual processes with a centralized digital platform capable of enforcing structured, transparent, and auditable workflows across geographically distributed teams.

1.3 Solution Overview

FACTS is a three-tier web application built on Node.js (Express), React, and MySQL, providing:

  • End-to-end case lifecycle management from pre-screening through closure, with configurable multi-stage approval pipelines
  • ITS (Institutional Information System) integration for automated applicant identity retrieval and photo verification
  • Eight-section digital counseling forms replacing paper-based counseling documentation
  • Multi-level executive approval hierarchy with up to 5 approval levels, rework and rejection flows
  • Payment schedule management tracking Qardan Hasana and Enayat disbursements with installment tracking
  • Dynamic fund utilization reporting with a configurable form builder for quarterly reassessments
  • Automated PDF generation for cover letters and quarterly reports via PDFKit and Puppeteer
  • Granular Role-Based Access Control (RBAC) with Jamiat/Jamaat organizational scope isolation
  • Real-time in-app notifications for case status changes, assignments, and SLA alerts
  • SLA tracking with automated breach detection and warning alerts per workflow stage

5.2 Architecture Pattern

FACTS follows a decoupled three-tier architecture:

  • Client Tier: React 18 Single Page Application (SPA) consuming a stateless JSON REST API. All data is managed via React Query with automatic caching and optimistic updates. Auth state is managed by a global React Context (AuthContext).
  • API Gateway Tier: Node.js / Express server exposing 25 route modules with ~150+ endpoints. JWT middleware enforces authentication; a RBAC middleware enforces resource-level permissions and Jamiat/Jamaat scope on every request. Business logic is encapsulated in service and utility files.
  • Data Tier: MySQL 8.0 InnoDB database with 50+ normalized tables, enforced foreign key constraints, and parameterized queries throughout. File uploads are stored on local filesystem under uploads/.

7. Functional Requirements

7.1 Authentication & Authorization Module

FR-AUTH-001: User Login

Field Detail
Priority Critical
Description The system shall authenticate users via username/email and password, returning a signed JWT token valid for 24 hours
Acceptance Criteria User can log in with username or email; incorrect credentials return 401; successful login returns JWT + user object with role and permissions; token is validated on every subsequent protected API request

Business Rules:

  • Passwords are stored as bcrypt hashes; password_hash() used at creation, bcrypt.compare() at login
  • JWT payload contains userId, username, role, and executive_level
  • User is_active flag is checked on every authenticated request — inactive users are rejected mid-session
  • Login endpoint is rate-limited to prevent brute-force attacks
  • No self-service password reset; admin must reset passwords

FR-AUTH-002: JWT Session Management

Field Detail
Priority Critical
Description The system shall enforce JWT-based stateless sessions for all protected API routes
Acceptance Criteria Requests without a valid Bearer token receive 401; expired tokens receive 401; logout is client-side token discard; profile and permissions retrievable from GET /api/auth/profile

FR-AUTH-003: Role-Based Access Control

Field Detail
Priority Critical
Description The system shall enforce granular RBAC at the API route level using a resource:action permission model
Acceptance Criteria Each protected endpoint checks the requesting user's role permissions against the required resource:action pair; insufficient permissions return 403; Super Admin bypasses permission checks; permissions are configurable per role via admin UI

FR-AUTH-004: Jamiat/Jamaat Scope Isolation

Field Detail
Priority Critical
Description The system shall filter data results based on the authenticated user's assigned Jamiat/Jamaat organizational scope
Acceptance Criteria Approval roles (Welfare, ZI) see only cases from their assigned Jamiat(s); filling roles (Counselors) see only cases from their assigned Jamaat(s); Super Admin and Admin have no scope restriction; scope is enforced at the database query level, not just UI filtering

FR-AUTH-005: Multi-Role User Support

Field Detail
Priority High
Description Users may be assigned multiple roles with different Jamiat/Jamaat scopes per role
Acceptance Criteria Roles stored in user_roles junction table with per-role Jamiat/Jamaat assignments; user can switch primary display role via PATCH /api/users/me/primary-role; system resolves effective permissions from all assigned roles

FR-CASEID-001: Create Case Identification

Field Detail
Priority Critical
Description Authorized users shall create pre-assessment eligibility screening records for potential welfare applicants
Acceptance Criteria Required: its_number, eligible_in (case type), total_family_members (0–25), earning_family_members (0–20), individual_income (0–9,999,999), family_income (0–9,999,999); applicant data auto-populated via ITS number lookup; duplicate ITS check blocks re-registration; status defaults to 'pending'

FR-CASEID-002: ITS Number Auto-Lookup

Field Detail
Priority Critical
Description The system shall auto-populate applicant demographic data by querying the external ITS API when an ITS number is entered
Acceptance Criteria Fields populated: full_name, age, gender, phone, email, photo (base64), address, jamiat, jamaat; lookup first checks local DB (applicants table), then falls back to external ITS API; failure returns graceful empty response; existing ITS numbers flagged for duplicate detection

Field Detail
Priority Critical
Description Authorized reviewers shall mark case identification records as eligible or ineligible
Acceptance Criteria Permission: case_identification:approve; marking 'eligible' auto-creates a Case record with an auto-generated case number (e.g., BS-0001); marking 'ineligible' stores review_remarks and closes the identification; reviewer and timestamp recorded; notification sent on status change

FR-CASEID-004: Edit Assessment Fields

Field Detail
Priority High
Description Authorized editors shall modify income and assessment fields on pending or ineligible case identification records
Acceptance Criteria Permission: case_identification:edit; editable fields: individual_income, family_income, eligible_in, total_family_members, earning_family_members, remarks; all income field changes logged to case_identification_income_logs with old/new values and editor ID for full audit trail

FR-CASEID-005: Revert Ineligible to Pending

Field Detail
Priority Medium
Description Authorized editors shall revert an ineligible case identification back to pending status for re-review
Acceptance Criteria Only ineligible records can be reverted; status changes to 'pending'; review fields (reviewed_by, reviewed_at, review_remarks) cleared; audit log entry created

FR-CASE-002: Case Assignment

Field Detail
Priority Critical
Description DCMs/Admins shall assign cases to counselors with Jamiat/Jamaat scope filtering
Acceptance Criteria Available counselors endpoint filters by role and optional Jamiat/Jamaat; assignment stored in cases.assigned_counselor_id; DCM stored in cases.roles; notification sent to assigned counselor on assignment

FR-CASE-003: Case Workflow Actions

Field Detail
Priority Critical
Description Cases shall progress through configurable workflow stages via approve, reject, rework, and resubmit actions
Acceptance Criteria Each action validates user's role permission against the current workflow stage; approve advances case to next stage; reject records reason and blocks case; rework returns case to previous stage; resubmit re-advances a reworked case; all actions write to status_history and workflow_logs; stage entry timestamps recorded in cases.current_stage_entered_at for SLA calculation

FR-CASE-004: Case Closure

Field Detail
Priority High
Description Authorized users shall close completed cases by uploading closure documentation
Acceptance Criteria Permission: cases:close_case; requires closure reason and 1–10 file uploads (PDF, DOC, DOCX, XLS, XLSX, JPG, PNG; max 10MB each); creates case_closures and case_closure_documents records; case status set to 'closed'

FR-CASE-005: Case Comments

Field Detail
Priority High
Description Authorized users shall add general and workflow-specific comments to cases
Acceptance Criteria General comments stored in case_comments; workflow-step-specific comments stored in workflow_comments; executive comments support is_visible_to_dcm flag and executive_level tag; comment type options: general, rejection, approval, note, feedback

FR-CASE-006: Duplicate ITS Detection

Field Detail
Priority High
Description The system shall enforce one active case per ITS number and flag duplicates
Acceptance Criteria Check: GET /api/cases/its/:itsNumber/active returns active cases for the ITS; if an active case exists, new case creation is blocked or flagged with is_duplicate = 1; duplicate cases shown in case list with visual flag

FR-CASE-007: Case Filtering and Pagination

Field Detail
Priority High
Description Users shall filter and paginate case lists based on multiple criteria
Acceptance Criteria Filter parameters: status, case_type, case_type_id, assigned DCM, assigned counselor, search (name/ITS/case number), jamiat_id, jamaat_id, current_workflow_stage_id, is_duplicate; results scoped by user's Jamiat/Jamaat assignment; SLA status (on_time/at_risk/breached) calculated and returned per case

9.2 Request Lifecycle

1. React SPA makes Axios HTTP request with Bearer token
       │
2. Express receives request
   → Helmet adds security headers
   → CORS validates origin
   → Rate limiter checks request frequency
       │
3. JWT Auth Middleware:
   - Decodes and verifies JWT token
   - Checks user.is_active in DB
   - Attaches user object to req.user
   - Rejects with 401 if invalid / expired
       │
4. RBAC Permission Middleware:
   - Checks req.user.role against required resource:action
   - For scoped roles: applies Jamiat/Jamaat filters to query
   - Rejects with 403 if insufficient permission
       │
5. Joi Validation:
   - Validates request body / query params against schema
   - Returns 400 with field-level errors on failure
       │
6. Route Handler (Business Logic):
   - Executes parameterized MySQL queries via connection pool
   - Calls services (emailService, notificationService, slaCalculator)
   - Invokes PDF/Excel generators as needed
   - Handles file I/O via Multer
       │
7. Response:
   - Returns JSON payload with appropriate HTTP status
   - React Query caches and updates UI state
   - Toast notification displayed to user

10.7 Key Design Decisions

  1. InnoDB engine — All tables use InnoDB for foreign key constraint enforcement and ACID transactions
  2. Organizational scoping at SQL level — Jamiat/Jamaat filters injected into queries based on user_roles.jamiat_ids / jamaat_ids columns
  3. One-active-case-per-ITS enforcement — Utility activeCasePerIts.js prevents duplicate active cases for the same applicant
  4. JSON columns — workflow_history, associated_statuses, responses_json stored as JSON/longtext for flexibility
  5. Soft delete pattern — Workflow stages and checklist items use is_active flag instead of hard deletion
  6. Self-referencing FK — workflow_stages.next_stage_id and fund_utilization_questions.parent_question_id implement linked-list stage chains and conditional form logic
  7. Base64 photo storage — Applicant and user photos stored as Base64 longtext in DB (avoids filesystem dependency for profile images)
  8. Decimal precision — All financial fields use DECIMAL(15,2) for currency accuracy

12.2 SLA Calculation Flow

Case moves to new workflow stage
       │
       ├── Record current_stage_entered_at = NOW()
       │
SLA Check triggered (manual POST /check-sla or event-based)
       │
       ├── For each case in non-terminal status:
       │       elapsed = NOW() - current_stage_entered_at
       │       stage.sla_value / sla_unit → convert to hours
       │       stage.sla_warning_value / sla_warning_unit → convert
       │
       ├── If elapsed ≥ sla_value:
       │       status = "breached"
       │       record sla_breached_at
       │       notify: DCM + relevant approver + admin (error notification)
       │
       ├── If elapsed ≥ sla_warning_value (but < sla_value):
       │       status = "at_risk"
       │       notify: DCM + relevant approver (warning notification)
       │
       └── Else: status = "on_time"
           (returned as sla_status field on case list API)

12.3 ITS Auto-Fetch Flow

User enters ITS number in Case Identification / Applicant form
       │
       ├── Frontend: GET /api/applicants/fetch-from-api/:itsNumber
       │
       ├── Backend checks local DB first (applicants table)
       │       If found → return with source: "database"
       │
       ├── If not in DB → call external ITS API endpoint
       │       Parse response → map to applicant schema
       │       Return with source: "api"
       │
       ├── Frontend auto-populates form fields:
       │       full_name, age, gender, phone, email, photo,
       │       jamiat_name, jamaat_name, address
       │
       └── User reviews, edits if needed, then saves record

16. Security Requirements

16.1 Authentication Security

Requirement Implementation
Token-based auth JWT signed with JWT_SECRET (min 32 chars); expires in 24 hours
Password hashing bcryptjs — no plain-text passwords stored or logged
Active status check users.is_active verified on every authenticated request; deactivated users immediately lose access
Login rate limiting express-rate-limit on POST /api/auth/login to prevent brute-force
Default credential warning Admin must change default password (admin/password) on first login

16.2 API & Data Security

Requirement Implementation
SQL injection prevention All DB queries use mysql2 parameterized statements; no string concatenation in SQL
Input validation Joi schemas validate all request bodies before any DB access
HTTP security headers Helmet.js sets Content-Security-Policy, X-Frame-Options, HSTS, etc.
CORS restriction Only FRONTEND_URL env var origin accepted; credentials: true
Scope isolation Jamiat/Jamaat IDs from user_roles injected as SQL WHERE IN clauses for all non-admin queries
RBAC enforcement Every route uses permissionMiddleware checking resource:action against user's role permissions

16.3 File Upload Security

Requirement Implementation
MIME type validation Multer fileFilter validates MIME types; extension spoofing rejected
File size limit 10 MB per file via Multer limits.fileSize and Nginx client_max_body_size
Upload count limits Max 10 files per multi-upload request; max 10 closure documents per case
Secure file paths Uploaded files stored with random UUID-based names; original filenames sanitized and stored in DB only
Download access control All file download endpoints require authenticated JWT; case attachment downloads verify case access

16.4 Infrastructure Security

Requirement Implementation
HTTPS enforcement Nginx configured for SSL with HTTP → HTTPS redirect
Firewall Only ports 22 (SSH), 80 (HTTP redirect), 443 (HTTPS) open externally
Environment secrets All credentials in .env file; .gitignore excludes .env from version control
Database access MySQL bound to localhost only; no external DB port exposure
Error masking Production errors logged server-side; generic message returned to client
